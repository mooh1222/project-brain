# stale 자동화 — B+C 검수 모델에 코드변경 트리거 잇기 (설계 + 보류)

> **상태(2026-06-25): 설계 확정, 구현 보류.** A(추출·해소)는 이미 구현됨. B(자동화)는 착수 방아쇠 전까지 미룸.
> 이 문서는 여러 세션에 걸쳐 정렬한 결정을 보존한다(소실 방지). 구현 시 TDD·결정론 원칙은 [CLAUDE.md] 따른다.

**Goal:** "코드가 바뀌면 어느 적재 지식이 낡았나"를 찾고 해소하는 흐름을, 새 철학이 아니라 **이미 있는 검수 모델(B+C)을 "코드 변경" 트리거에 연결**하는 것으로 설계한다. 완전 자동(사람 무개입) supersede는 하지 않는다.

---

## 1. 현재 상태 — A는 이미 있다

코드변경→갱신 흐름의 **추출과 해소는 이미 구현·검증됨**(2026-06-25 bb2 실코퍼스 실행 확인):

- `stale-check` — 앵커 코드(`path`)가 기록 `commit_sha` 이후 바뀐 매핑을 `candidates`(mapping_key + 바뀐 locator + 변경유형 M/A/D/R)로 추출. 읽기 전용.
- `mark-checked` — "의미 그대로"로 검토 끝낸 매핑의 locator `commit_sha`를 갱신(해소). 쓰기.

점검·진단 명령 계열(README 참고): `lint`(끊긴 참조=아웃바운드), `graph isolated`(고립=인바운드), `stale-check`(코드변경), `doctor`(환경). stale-check이 "갱신 필요 요소 추출" 명령이다 — 새로 만들 게 아니라 이미 있다.

**그래서 A로 새로 짤 코드는 없다.** 남은 자동화 가치는 전부 아래 B에 있다.

## 2. 검수 철학 — B+C 하이브리드 (이 자동화의 틀)

정본: `design-canonical.md §2`, `docs/specs/2026-06-05-bb2-brain-bc-review-model-design-v2.md`.

- **B(자동)**: 검증된 근거 + 적대검증 통과 + lint clean이면 사람 손 없이 `reviewed`로 적재.
- **모호는 안 삼킴**(B 전용): 근거 약함·충돌·적대검증 실패는 `candidate`로.
- **C(노출+lazy promote)**: candidate를 query 시점에 "확인 필요" 라벨로 노출 → 사용자가 그 자리에서 promote/수정. (최초 이진 설계는 candidate 미노출=silent였고 C가 고침, spec v2 §4.1.)
- **사용자 1인 전수 검수는 병목이라 안 한다** — 사람 역할 = "예외 판정 + 사용 시점 수정".

> 주의: "완전 자동 = 코퍼스 오염"이 아니다. 가드는 "사람 전수검수"가 아니라 **"근거 + 적대검증 게이트"**다. 확실하면 자동, 모호하면 candidate+사용자.

## 3. stale를 B+C에 잇기 (B 설계)

코드가 바뀌면(예: develop pull) stale-check이 짚은 영향 매핑 각각을 에이전트가 diff+매핑 본문을 읽고 판단:

- **갈래 1 — 확실하면 자동(B 논리)**
  - *그대로인 게 확실*(앵커 함수 불변, 옆 함수만 변경): 앵커 `commit_sha`만 현재로 갱신(=mark-checked). 사람 개입 0.
  - *바뀐 게 확실*: 새 의미를 재도출 + **적대검증 통과 시** supersede 리비전 작성(옛 버전 **이력 보존** — 덮어쓰기 아님). 사람 개입 0. ← 더 공격적, 후순위.
- **갈래 2 — 모호하면 query에서(C 논리)**
  - 덮어쓰지 않고 매핑에 "코드 변경됨 — 확인 필요" 표식만. reviewed 답은 계속 서빙(advisory 부착).
  - 그 매핑이 답에 뜰 때 사용자가 promote(유효)/수정/supersede 판정.

핵심: **새 메커니즘이 아니라 B(자동)·C(query 노출)를 "코드 변경" 트리거에 연결**하는 것.

## 4. 정밀화는 엔진 파서가 아니라 에이전트 몫 (Option C 제외)

거짓 경보(파일 단위라 같은 파일 무관 변경도 다 걸림)를 줄이는 정밀화의 **올바른 자리는 갈래 1의 에이전트**다:

- 줄번호를 제거했으므로 엔진은 "바뀐 hunk → 어느 symbol"을 싸게 못 잇는다(symbol 위치 재파싱 필요). 반면 에이전트는 diff hunk + symbol 이름을 읽고 "이 변경이 이 함수 건드렸나"를 **실제 코드로** 판단 — 진짜 정밀(휴리스틱 아님).
- 즉 **B(에이전트 diff 읽기)가 정밀화의 실익을 가져온다.** 엔진 symbol 파서는 그 위 비용 최적화일 뿐.
- **데이터 근거(bb2, 2026-06-25)**: CodeLocator 594개 중 **84%가 같은 path 공유**(hot file: DisturbManager.cpp 29개 등). 성장+churn하면 악화. 그래도 엔진 정밀화는 감지 단계 **drop-in 교체**라 정말 필요할 때(hot 파일 매핑 수백 개로 에이전트 토큰비용이 아플 때) 무손실로 추가 가능.
- 비용 메모: 굳이 엔진에 넣어도 git에 cpp·java·objc funcname 드라이버 내장(`git log -L`)이라 파서를 새로 짤 필욘 없지만, B가 같은 일을 더 정확히 하니 moot.

## 5. 미머지 앵커 처리 (실코퍼스 발견)

**발견(2026-06-25)**: PR 머지 전에 적재하면 앵커 `commit_sha`에 **작업 브랜치 커밋**이 박힌다. 그 커밋은 develop에 없어, stale-check이 `작업커밋..origin/develop`을 비교하며 파일을 "삭제(D)"/"변경(M)"으로 **거짓 신호**를 낸다.

- 산 증거: BoostedBomb 앵커 13개 전부 `commit_sha=b27a23e385` = `feature/5.6/disturb-boostedbomb` 커밋(develop 조상 아님, 파일 develop 미존재). stale-check의 D 3건이 전부 이 아티팩트(실제 삭제 아님).
- **머지 방식 의존**: bb2 develop은 머지 커밋(`Merge pull request #NNNN`)과 직접 푸시가 혼재하지만 **둘 다 원커밋 해시를 보존**(스쿼시·리베이스 아님 — 직접 푸시 존재가 그 방증) → 머지되면 b27a23e가 develop 조상이 됨 → **"정정" 불필요, 거짓 신호 자동 해소**. (스쿼시·리베이스 팀이면 새 해시라 정정 필요 — bb2는 해당 없음.)
- **B 요건**: stale-check이 **"앵커 커밋이 develop 조상 아님(=미머지)"** 을 D/변경과 **별개 범주로 구분 표시**(차단 아니라 라벨). 안 그러면 머지 전 적재가 전부 거짓 stale로 뜬다.

## 6. symbol 유지 / 줄번호 제거 (확정 근거)

- `path`=어느 파일 / `symbol`=무엇(이름) / `commit_sha`=어느 버전(변경감지 기준점) / 줄번호=불안정 좌표(제거).
- symbol은 변경감지엔 안 쓰지만(`path+commit_sha`로 함) **검색 색인(`surface.py` path+symbol)·표시(answer/show/search/graph)·갈래1 정밀화 입력**으로 필요. → 유지.
- 줄번호: 되살리지 않음(영구), 기존 데이터에서 굳이 지우지도 않음(안 읽혀 무해), 신규는 build가 안 만듦.

## 7. 착수 방아쇠 / 안 하는 것

- **방아쇠**: stale-check 결과를 손으로 triage하는 게 실제로 거슬릴 때, 또는 query 시점 노출이 필요해질 때. 그 전엔 `stale-check` 수동 실행으로 충분.
- **안 한다**: 완전 자동 supersede(사람·적대검증 무개입), 자동 hook, 엔진 symbol 파서(정밀화는 에이전트가).

## 8. 구현 순서 (방아쇠 후)

1. stale-check에 "미머지 앵커" 구분 라벨 추가(§5) — 엔진, 합성 테스트.
2. 갈래 2(query 노출): stale-set를 query/show가 읽어 "확인 필요: 코드 변경" advisory 부착. (캐시 설계 — query마다 git 금지.)
3. 갈래 1(에이전트 트리아지): 스킬/플로우로 diff+매핑 읽고 확실-불변 자동 갱신/변경 supersede 초안. 실코퍼스 회귀 필요.
